SSH Basics

TermMeaning
SSHSecure Shell — encrypted remote login protocol
sshdSSH daemon — the SERVER side service
sshSSH client — used to CONNECT to a server
PortDefault port 22 (TCP)
Public keyShared freely — server stores it
Private keySecret — never share! Stays on client
known_hostsStores trusted server fingerprints
authorized_keysStores allowed client public keys
Rule: Private key stays on YOUR machine. Public key goes on the REMOTE server. Never share the private key!
🚀

Connecting with SSH

$ ssh server01 # connect as current user
$ ssh ali@server01 # connect as user ali
$ ssh ali@192.168.1.10 # connect using IP address
$ ssh -p 2222 ali@server01 # connect on custom port
$ ssh ali@server01 hostname # run ONE command remotely, then exit server01.example.com
$ ssh ali@server01 'ls /var/log' # run remote command with quotes
$ ssh -X ali@server01 # X11 forwarding (GUI apps remotely)
$ ssh -v ali@server01 # verbose — debug connection issues
First time connecting — host verification The authenticity of host 'server01' can't be established.
RSA key fingerprint is SHA256:xxxxxxxxxxx
Are you sure you want to continue connecting (yes/no)? yes
Warning: Permanently added 'server01' to known_hosts.
Always verify the fingerprint on first connect — this prevents man-in-the-middle attacks. After typing yes, the key is saved to ~/.ssh/known_hosts.

SSH Key-Based Authentication — How It Works

Step by Step Setup
# STEP 1 — Generate key pair on CLIENT $ ssh-keygen Generating public/private rsa key pair. Enter file: /home/ali/.ssh/id_rsa ← default Enter passphrase: (optional but recommended) Your public key: ~/.ssh/id_rsa.pub Your private key: ~/.ssh/id_rsa
# STEP 2 — Copy public key TO the server $ ssh-copy-id ali@server01 Number of key(s) added: 1 Now try logging into server: ssh 'ali@server01'
# STEP 3 — Login WITHOUT password! $ ssh ali@server01 [ali@server01 ~]$ ← no password asked! ✔
Key Files — Where They Live
FileLocationKeep Secret?
id_rsa~/.ssh/id_rsa✔ YES — private key
id_rsa.pub~/.ssh/id_rsa.pubNo — public key
authorized_keys~/.ssh/authorized_keysOn SERVER — holds allowed public keys
known_hosts~/.ssh/known_hostsOn CLIENT — trusted server fingerprints
config~/.ssh/configClient shortcuts & settings
# generate stronger ED25519 key (recommended) $ ssh-keygen -t ed25519 # produces: id_ed25519 and id_ed25519.pub
# manual copy if ssh-copy-id not available $ cat ~/.ssh/id_rsa.pub | ssh ali@server01 'cat >> ~/.ssh/authorized_keys'

~/.ssh/config — Client Shortcuts

Without config file
# have to type this every time: $ ssh -p 2222 -i ~/.ssh/mykey ali@192.168.1.100
With config file
# just type this: $ ssh myserver
~/.ssh/config Host myserver
    HostName 192.168.1.100
    User ali
    Port 2222
    IdentityFile ~/.ssh/mykey

Host webserver
    HostName web.example.com
    User root
    IdentityFile ~/.ssh/id_rsa
$ chmod 600 ~/.ssh/config # must be readable only by owner!
🛡️

sshd Server Config

/etc/ssh/sshd_config Port 22 # change to non-standard port
PermitRootLogin no # disable root login!
PasswordAuthentication no # keys only
PubkeyAuthentication yes # allow key login
AllowUsers ali sara # whitelist users
MaxAuthTries 3 # max failed attempts
ClientAliveInterval 300 # keepalive seconds
X11Forwarding no # disable GUI forwarding
$ systemctl reload sshd # apply config changes (no disconnections)
$ sshd -t # test config for syntax errors BEFORE reloading
Always run sshd -t before reloading sshd — a bad config can lock you out of the server!
📦

scp & sftp — Secure File Transfer

scp — Secure Copy
# copy file FROM local TO remote $ scp file.txt ali@server01:/tmp/
# copy file FROM remote TO local $ scp ali@server01:/tmp/file.txt .
# copy entire directory (-r recursive) $ scp -r mydir/ ali@server01:/home/ali/
# copy on custom port $ scp -P 2222 file.txt ali@server01:/tmp/ # NOTE: capital -P for scp (lowercase for ssh)
sftp — Interactive Transfer
$ sftp ali@server01 sftp> ls # list remote files sftp> lls # list LOCAL files sftp> get file.txt # download file sftp> put file.txt # upload file sftp> cd /tmp # change remote dir sftp> lcd /home # change local dir sftp> bye # exit

SSH File Permissions

SSH is very strict about permissions — wrong permissions = key ignored or connection refused!
File / DirRequired PermissionCommand
~/.ssh/700 (drwx------)chmod 700 ~/.ssh
~/.ssh/id_rsa600 (rw-------)chmod 600 ~/.ssh/id_rsa
~/.ssh/id_rsa.pub644 (rw-r--r--)chmod 644 ~/.ssh/id_rsa.pub
~/.ssh/authorized_keys600 (rw-------)chmod 600 ~/.ssh/authorized_keys
~/.ssh/config600 (rw-------)chmod 600 ~/.ssh/config
# fix all SSH permissions at once: $ chmod 700 ~/.ssh $ chmod 600 ~/.ssh/id_rsa ~/.ssh/authorized_keys $ chmod 644 ~/.ssh/id_rsa.pub

Managing sshd Service

$ systemctl status sshd # check if sshd is running
$ systemctl enable --now sshd # start + enable at boot
$ systemctl reload sshd # reload config (no disconnections)
$ systemctl restart sshd # restart (drops active connections!)
$ ss -tlnp | grep ssh # verify sshd is listening on port 22 LISTEN 0 128 0.0.0.0:22 0.0.0.0:* users:(("sshd",pid=1234))
$ journalctl -u sshd -f # watch sshd logs live

Exam Tips

⚠ LOCKOUTAlways run sshd -t before systemctl reload sshd — a bad config can lock you out of the server permanently!
🔑 KEYSPrivate key id_rsa must be 600. If permissions are too open, SSH ignores the key silently and falls back to password.
✔ COPYUse ssh-copy-id — it handles creating ~/.ssh/, authorized_keys, and correct permissions automatically.
📌 PORTscp uses -P (capital) for port. ssh uses -p (lowercase). Easy to mix up in the exam!
⚠ ROOTSet PermitRootLogin no in sshd_config — never allow direct root SSH login in production or exam scenarios.
💡 RELOADUse reload not restart for sshd — reload applies config without dropping active SSH connections.
✔ ED25519ssh-keygen -t ed25519 generates a stronger, shorter key than RSA. Preferred on modern RHEL systems.
🔍 DEBUGSSH not connecting? Use ssh -v ali@server for verbose output — shows exactly where authentication fails.
